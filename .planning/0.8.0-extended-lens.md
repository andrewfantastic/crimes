## 1. Product framing

The 0.3.0â€“0.7.5 arc built out structural, IA, duplication, frontend,
and calibration coverage. Reviewing the bundle against the broader
static-analysis ecosystem (ESLint, Biome, Knip, typescript-eslint,
Semgrep, SonarQube) and against the kinds of bugs that recur in
real-world code surfaced four families where **no mainstream tool
catches the pattern**, the pattern correlates with concrete change
risk or agent risk, and a deterministic heuristic looks feasible.

The 0.8.0 thesis:

> Extend the lens to cover four families of "common-sense" crimes
> that linters don't catch and humans / agents repeatedly ship:
> timezone-and-date crimes, narrow naming-tier crimes, hot-path I/O
> crimes, and asset-weight crimes.

Each family below is justified by a what-already-exists audit
(Â§5.aâ€“d) so we don't drift into linter territory.

## 5. Research questions per family

Each family below lists the candidates we're researching, the
specific evidence we need from the codebase + ecosystem before
committing, and the eval scenarios that would prove the detector is
useful and not noisy.

### 5.a Date/time crimes

Candidates: `timezone_unsafe_parse`, `mixed_utc_local_methods`,
`locale_drift`, `dst_naive_arithmetic`, `date_string_concat`,
`date_equality_misuse`.

Research questions:

1. **Heuristic feasibility.** For each candidate, write a one-page
   detection heuristic against the existing `language-js` AST
   layer. Confirm what the parser already exposes (Date call sites,
   method names per Date instance, ISO string literals) and what
   would need to be added. We have `parsed.dateNowOrNewDateUses`
   today â€” what else does `language-js/src/parse/` need to surface?
2. **False-positive surface on real repos.** Run a prototype of
   each detector on the crimes monorepo, the bundled
   `examples/messy-ts-app` fixture, and at least three external
   open-source TS/JS repos with non-trivial date handling
   (candidates: a calendar app, a scheduling library, a billing
   project). Record finding counts and the top false-positive class
   per detector.
3. **Coexistence with `direct_date`.** `direct_date` already flags
   `Date.now()` / `new Date()` in domain code. The new detectors
   should not double-count â€” `mixed_utc_local_methods`, for
   example, should fire on a Date already created elsewhere, not
   on a `new Date()` that `direct_date` would already flag.
4. **Severity calibration.** Initial intuition: most date crimes
   default `low` or `medium`, but `mixed_utc_local_methods` and
   `dst_naive_arithmetic` may justify `high` because the bug class
   is silent in tests that don't cross a DST boundary.

Eval scenarios needed (one per detector, all five scenario kinds):

- **bugfix** scenario: agent must identify a timezone-introduced
  bug from `crimes scan` output before proposing the fix.
- **review** scenario: PR diff that introduces a `new Date(string)`
  parse, expecting the verdict to flag the new high.
- **refactor**, **context**, **plan** scenarios per the pattern
  established in `evals/scenarios/`.

Eval prerequisite: extend `examples/messy-ts-app` with intentionally
buggy timezone code, similar to the IA expansion done in 0.7.5.

### 5.b Naming-tier crimes

Candidates: `singular_plural_type_mismatch`, `boolean_naming_drift`,
maybe-scope `exported_stop_word_name`.

Research questions:

1. **TypeScript type-info dependency.** `singular_plural_type_mismatch`
   needs type info, not just syntax â€” `users: User` vs `user: User[]`
   only works if the parser knows the annotated type. Confirm
   `language-js` exposes type annotations or commit to using
   `ts-morph`'s type checker. Performance budget matters: a
   type-checking detector is much slower than the others.
2. **Pluralisation library scope.** English plural rules are
   nuanced (`status`/`statuses`, `criterion`/`criteria`,
   `data`/`data`). Will we ship a built-in pluraliser or accept
   noise on edge cases? Likely answer: small built-in heuristic +
   conservative confidence + exemption config to silence specific
   names.
3. **Boolean naming false positives.** What about `loading`,
   `ready`, `disabled`, `active`? They're booleans without
   `is/has/should` prefix and they're idiomatic React state. Need a
   built-in exemption list **plus** the new `detectors.options`
   user override.
4. **`stop_word_name` agent-risk angle.** Does scoping to *exported*
   identifiers (where naming actually misleads downstream code)
   produce a useful signal that ESLint's `id-denylist` does not?
   Hypothesis: yes â€” `id-denylist` is global; our angle is "the
   name is a stop word **and** it's exported across module
   boundaries where the loss of context is highest."

Eval scenarios needed: variants per scenario kind, but
calibration-heavy. The eval harness should specifically test that
React state hooks don't trip `boolean_naming_drift`.

### 5.c Hot-path & portability crimes

Candidates: `sync_io_in_hotpath`, `hardcoded_local_path`,
`hardcoded_localhost`. Maybe-scope: `catch_rethrow_no_context`,
agent-risk `dead_export`.

Research questions:

1. **Hot-path definition.** The 0.4.0 shape system already
   classifies functions as `route_handler`, `react_component`,
   `page_export`, `cli_command_registrar`, `test_callback`,
   `domain`, `unknown`. `sync_io_in_hotpath` fires when
   `readFileSync` / `writeFileSync` / `existsSync` / etc. appear
   inside a non-test, non-CLI hot-path shape. Confirm shape
   classification accuracy on real repos before relying on it.
2. **`hardcoded_local_path` scope.** Patterns like `/Users/<name>/`,
   `C:\\Users\\`, `/home/<name>/` are clear. What about
   `/tmp/...`? Configurable threshold or always-skip?
3. **`hardcoded_localhost` scope.** Should skip config files,
   docker-compose, `.env.example`, scripts under `scripts/`,
   anything in `test/` or `examples/`. Need a robust exclusion
   model.
4. **Dead export with agent-risk framing.** Specifically: a dead
   export whose name is within edit distance 1 of a live export
   (e.g., `getUserById` exported but unused, `getUserByID` is the
   live one). Agents importing by name are likely to pick wrong.
   This is a *narrow* angle Knip doesn't have. Worth a prototype.
5. **`catch_rethrow_no_context` framing.** ESLint's
   `no-useless-catch` catches `try { â€¦ } catch (e) { throw e; }`.
   Our angle: catches that rethrow with no logging, no context
   wrap, and no error chaining â€” silently swallowing the stack
   trace location is the actual harm. Worth research only if the
   pattern is common enough to matter.

Eval scenarios needed: scenarios per kind for each promoted
detector, with explicit shape-correctness tests for
`sync_io_in_hotpath`.

### 5.d Asset crimes

Candidates: `oversized_raster`, `raster_should_be_vector`,
`svg_with_embedded_raster`.

Research questions:

1. **File-system scope.** `crimes` today walks JS/TS source. Asset
   detectors need to walk `*.{png,jpg,jpeg,gif,webp,avif,svg}` too.
   Confirm this fits the existing `fast-glob` pipeline and that
   the default `include` pattern + per-file extensions extends
   cleanly.
2. **Threshold defaults.** What counts as "oversized"? Common
   industry guidance: hero images under 200KB, content images
   under 100KB, icons under 10KB. Defaults should be tunable via
   `thresholds.assetWeight` in `crimes.config.json`.
3. **`raster_should_be_vector` heuristic.** PNG/JPG with width AND
   height â‰¤ 64px â†’ strongly suggests an icon. Read image
   dimensions without decoding the full image (the PNG/JPG/WebP
   headers are enough). Confirm a zero-dependency or
   tiny-dependency way to do this.
4. **`svg_with_embedded_raster` heuristic.** SVG containing
   `<image href="data:image/png;base64,...` or
   `<image xlink:href="data:image/jpeg;base64,...` defeats the
   purpose of using SVG. Pure regex / string scan, no SVG parser
   needed. Confirm this is a real recurring pattern (likely
   triggered by Figma/Sketch SVG exports).
5. **Default excludes.** Asset detectors should default-exclude
   `node_modules`, `dist`, `build`, `coverage`, `**/__snapshots__`,
   `**/.git`, `**/public/vendor`, plus probably `**/fixtures/**`
   if we don't want test fixtures to dominate. Confirm against the
   existing `exclude` defaults.

Eval scenarios needed: at least one fixture file in
`examples/messy-ts-app` per detector â€” an oversized PNG, a
4Ă—4-pixel PNG that should be an SVG, an SVG containing an embedded
raster. Care must be taken to keep the fixture's git footprint
small.

## 6. Per-detector exemption config (`detectors.options`)

Today suppression has two tiers:

1. **Detector toggle** â€” `detectors.disable: ["X"]` kills detector
   X everywhere.
2. **Fingerprint suppression** â€” `crimes ignore <fingerprint>`
   kills one specific finding by `<type>::<file>::<symbol>`.

The new naming-tier detectors need a third tier: **per-detector
exempt values**. If `data` is the right name for a particular
codebase, we want to suppress the *value*, not all instances and
not the whole detector.

Proposed shape (zod-validated, additive, no schema bump):

```json
{
  "detectors": {
    "options": {
      "stop_word_name": {
        "allowedNames": ["data", "ctx"]
      },
      "boolean_naming_drift": {
        "allowedNames": ["loading", "ready", "active", "disabled"]
      },
      "magic_domain_literal_scatter": {
        "allowedLiterals": ["draft", "published"]
      },
      "name_behavior_mismatch": {
        "allowedFunctions": ["getOrCreateUser"]
      },
      "locale_drift": {
        "allowedLocales": ["default"]
      }
    }
  }
}
```

Each detector decides what option keys it consumes. The schema
validates keys per detector; unknown keys raise `ConfigParseError`
(same exit-2 path as the rest of config).

Design notes:

- **Built-in defaults coexist.** Each affected detector ships
  with a small built-in allowlist (e.g., `boolean_naming_drift`
  ships with `["loading", "ready", "active", "disabled",
  "pending", "open", "closed", "expanded", "collapsed"]`). User
  config is additive on top.
- **No fingerprint duplication.** `detectors.options` exemptions
  apply *before* findings are emitted, so they never reach the
  suppression layer at all. Fingerprint suppression still works
  for instance-level decisions (e.g., "this one specific call to
  `Date.now()` in `metrics.ts` is fine even though we're trying
  to reduce them overall").
- **Documented in `crimes init`.** The starter `crimes.config.json`
  written by `crimes init` gets a commented-out
  `detectors.options` block as a discovery aid.
- **No CLI command.** Unlike `crimes ignore`, exemption config
  lives in `crimes.config.json` (committed, reviewed). No
  `crimes exempt â€¦` command â€” the value lives in the file, edited
  by hand.

This change is small, additive, and should land **first** in the
0.8.0 implementation sequence (Â§7) so the naming-tier and locale
detectors can consume it from day one.

## 7. Eval-harness plan

The 0.7.5 harness already runs `claude` + `codex` CLIs against
12 scenarios across 5 kinds. The 0.8.0 expansion:

1. **Add scenarios per new detector.** Following the 0.7.5
   "Cluster 5: large_file detector coverage" pattern â€” one scenario
   per detector per scenario kind where it makes sense, plus
   negative scenarios where the detector should NOT fire.
2. **Variance sampling.** Use the existing `evals:variance`
   workflow on each new detector before release to separate agent
   inconsistency from real signal.
3. **Judge-model pass.** Opt-in `--judge` pass against each new
   detector's scenarios to catch open-ended quality regressions
   the structural rubric misses.
4. **Cross-detector regression.** Run the full 33-scenario
   existing suite after every detector lands; baseline patch bump
   (per the eval-version policy in
   [`evals/README.md`](../../evals/README.md)) on every calibration
   change.
5. **Multi-agent coverage.** Both `claude` and `codex` must pass
   each new scenario before the detector is considered ready. This
   is already the harness default â€” flagged here to confirm.

Open question: do we want to add a third agent (e.g., `aider`,
`continue`) to the harness during 0.8.0, or defer? Trade-off is
more coverage vs. setup complexity for users running evals
locally.

## Appendix A â€” ecosystem audit details

For each candidate, the exact existing tool that covers it:

- **Empty catch**: ESLint built-in `no-empty` with
  `{ allowEmptyCatch: false }`. Universal coverage in modern
  configs.
- **`await` in loop**: ESLint built-in `no-await-in-loop`. Known
  rule, often disabled because it has high false-positive rate â€”
  we'd inherit the same surface.
- **`@ts-ignore` discipline**: `@typescript-eslint/ban-ts-comment`
  on by default in `plugin:@typescript-eslint/recommended`.
- **`as any` proliferation**: `@typescript-eslint/no-explicit-any`
  in recommended.
- **Whole-library imports**: `eslint-plugin-import` rules
  `no-namespace`, `no-default-export`, plus per-package plugins
  like `eslint-plugin-lodash`.
- **Single-letter names**: `id-length` built-in.
- **Stop-word names** (basic): `id-denylist`, configurable but
  rarely set up. The agent-risk-exports variant is our angle.
- **Dead exports**: Knip and ts-prune. Both well-maintained and
  widely used.
- **Useless catch**: `no-useless-catch` built-in. Doesn't catch
  the no-context-wrap variant.

The pattern is consistent: where ESLint covers it, we don't
duplicate. Where the tool is niche or configurable-and-unused, we
revisit whether our agent-risk framing creates a sharper signal
than the linter angle.
