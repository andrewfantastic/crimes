## Investigation 1 ŌĆö language-js parser surface

**Where the existing parser stands today.** `packages/language-js`
exposes a single `parseFile()` that uses the TypeScript compiler API
(`ts.createSourceFile`) to produce a `ParsedFile` containing:

- `lineCount`
- `functions[]` (with 7-way `FunctionShape` classification)
- `dateNowOrNewDateUses[]` ŌĆö **only** `Date.now()` calls and
  `new Date(...)` creation sites; no method calls, no arithmetic,
  no string arguments
- `defaultExport`
- `navLiterals[]`, `uiStringLiterals[]`, `jsxElements[]`

Source: [`packages/language-js/src/parse/types.ts`](../packages/language-js/src/parse/types.ts)
and [`parse/dates.ts`](../packages/language-js/src/parse/dates.ts).

**Gap relative to 0.8.0 date detectors.** The current `DateUse`
shape (`{ kind: "now" | "new", line: number }`) is too narrow for
every date detector except `timezone_unsafe_parse`, and even that
one needs the argument captured. Specific deltas required:

| Detector | What language-js must expose |
|---|---|
| `timezone_unsafe_parse` | `new Date(arg)` with `arg.kind` (`string-literal`, `template`, `number`, `none`) and `arg.value` for string-literals |
| `mixed_utc_local_methods` | Every Date-instance method call (`getHours`/`getUTCHours`/etc.) with receiver identifier + method name + line |
| `locale_drift` | Every `.toLocaleDateString()` / `.toLocaleString()` / `.toLocaleTimeString()` call with arg count |
| `dst_naive_arithmetic` | Numeric literal arithmetic involving day/hour/minute constants (`86400000`, `* 60 * 60 * 1000`, etc.) at expression-statement level |
| `date_string_concat` | String + Date-method-result concatenation patterns |
| `date_equality_misuse` | Binary expressions (`==`/`===`/`!=`/`!==`) where one side is a Date-typed expression |

The parser does NOT need a full type checker for most of these ŌĆö
syntactic patterns suffice. Receiver tracking can be "best effort"
based on identifier name (variables named `date`, `d`, anything
returned from `new Date(...)` in the same scope) ŌĆö false-positive
control via confidence scores, not via type analysis.

The two detectors that genuinely benefit from type info ŌĆö
`mixed_utc_local_methods` and `date_equality_misuse` ŌĆö can ship
without it at lower confidence, then graduate to higher confidence
once we add type-checking (see Investigation 4 below).

**Implication for ┬¦5.a.1 + ┬¦9.1 (RESOLVED).** Yes, language-js
needs expansion. The expansion is bounded and additive ŌĆö new fields
on `ParsedFile`, no breaking changes to existing detectors. This is
phase 1 prerequisite work, lands before any date detector.

Concrete parser additions for phase 1:

```ts
// packages/language-js/src/parse/types.ts (additions)

export interface DateUse {
  kind: "now" | "new";
  line: number;
  // NEW: capture the argument to new Date(...) for parse-safety analysis
  argKind?: "none" | "string-literal" | "template" | "number" | "expression";
  argValue?: string;
}

export interface DateMethodCall {
  /** Receiver identifier as written, e.g. "date" in `date.getHours()`. */
  receiver: string;
  /** Method name, e.g. "getHours", "getUTCHours", "toLocaleDateString". */
  method: string;
  /** 1-based line. */
  line: number;
  /** Argument count (for locale-arg detection). */
  argCount: number;
}

export interface DateArithmetic {
  kind: "add" | "subtract";
  line: number;
  /** The numeric operand, e.g. 86400000. */
  operand: number;
  /** Whether the operand looks like a day/hour/minute/second multiplier. */
  unit?: "ms" | "second" | "minute" | "hour" | "day" | "week";
}

export interface ParsedFile {
  // ...existing fields...
  /** NEW: every Date-method call site. */
  dateMethodCalls?: DateMethodCall[];
  /** NEW: every Date-result arithmetic site. */
  dateArithmetic?: DateArithmetic[];
}
```

All additions are optional ŌĆö existing detectors and tests are
unaffected.

## Investigation 2 ŌĆö TS type info dependency

**Question (┬¦5.b.1, ┬¦9.2):** Does
`singular_plural_type_mismatch` (and possibly
`boolean_naming_drift`) need a TypeScript type checker, and if so,
do we accept a `ts-morph` dependency or use the raw `ts` API?

**Findings.**

Current parser uses `ts.createSourceFile` directly ŌĆö a single-file
syntactic parse, no type checker, no program graph. Type-aware
detectors need `ts.createProgram` + `program.getTypeChecker()`,
which requires:

1. The set of source files to include (we already discover these
   via `discoverFiles`).
2. A `CompilerOptions` object (we can synthesize a permissive
   default or read the project's `tsconfig.json`).
3. Significant memory overhead ŌĆö the TypeScript compiler holds the
   full project in memory.

**ts-morph vs raw `ts`:**

- **ts-morph**: wraps the TypeScript compiler with a friendlier
  object model. ~750KB, depends on `@ts-morph/common` + TypeScript.
  Adds a published-package dependency to the `crimes` CLI. Faster
  to write code against but the abstraction sometimes hides what
  the underlying API is doing (performance gotchas).
- **Raw `ts` API**: zero new dependency (TypeScript is already
  there). Verbose to use but identical performance and memory
  profile to ts-morph. Total code volume for the use cases we have
  (read variable annotations, inspect declared types) is moderate
  (~200-300 lines per detector).

**Recommendation: raw `ts` API.** Adding ts-morph for two
detectors is a significant dependency for a marginal ergonomic
win. We already use `ts` directly throughout `language-js`. Keep
the dependency surface tight.

**However**, do not require type checking for the *first cut* of
naming-tier detectors. The first cut works syntactically:

- `singular_plural_type_mismatch` v1: only fires on declarations
  where the type annotation is a **bare identifier** (`User`) or a
  **simple array** (`User[]`, `Array<User>`). No need to resolve
  the actual type ŌĆö just compare the surface form against the name.
- `boolean_naming_drift` v1: fires on declarations with explicit
  `: boolean` annotation or with a boolean-literal initializer
  (`false`/`true`/`!x`). Also no type checker needed.

This v1 path lands without expanding the parser into a full type
checker. Type-aware v2 (resolves type aliases, follows imports) is
a follow-up if v1's false-positive rate justifies it.

**Implication (RESOLVED ŌåÆ simpler than feared).** No ts-morph dep.
No type checker in phase 1. Syntactic parsing + careful confidence
scoring + the new `detectors.options` exemption config (┬¦6 in the
plan) gets us to a useful first-cut detector.

## Investigation 3 ŌĆö Asset file walking

**Question (┬¦5.d.1, ┬¦5.d.5):** Can the existing file-walking
pipeline support assets, and what's the cleanest architecture?

**Findings.**

`discoverFiles` in [`file-discovery.ts`](../packages/language-js/src/file-discovery.ts)
uses `fast-glob` with a configurable `include`/`exclude`. Default
`include` is `**/*.{ts,tsx,js,jsx,mjs,cjs}` ŌĆö does not match
assets today.

The whole detector pipeline assumes per-file: read source, parse
to `ParsedFile`, pass to detectors via `DetectorContext`. Asset
detectors don't need parsing and shouldn't trigger it.

**Two architectural options:**

1. **Extend the existing pipeline.** Add asset extensions to the
   default `include` list. Asset detectors check `extname(file)`
   and short-circuit if it's not an asset. Source files still get
   the full parse. Asset files get a stubbed `ParsedFile` (or the
   parse step is skipped for binary extensions).
2. **Separate asset pass.** Add an `AssetDetector` interface
   alongside `Detector`. The scanner does two discovery passes:
   one for source files (existing), one for assets (new). Asset
   detectors receive `{ file, absolutePath, stats }` ŌĆö no parsed
   AST.

**Recommendation: (2), separate asset pass.** Reasons:

- Asset detectors are conceptually distinct ŌĆö they answer "is
  this file too big / wrong format?" not "what's in this code?".
- Mixing parse-needing and non-parse detectors in one interface
  creates conditional code paths everywhere.
- The detector context surface for assets is small enough that a
  separate interface is cheaper than threading parsed-file
  optionality through every existing detector.

Sketch:

```ts
// packages/core/src/detector.ts (additions)

export interface AssetDetectorContext {
  file: string;
  absolutePath: string;
  /** Lazy: read only if the detector calls .read(). */
  read(): Promise<Buffer>;
  /** From fs.stat() ŌĆö pre-fetched. */
  byteSize: number;
  config: CrimesConfig;
}

export interface AssetDetector {
  id: string;
  name: string;
  description: string;
  whyItMatters: string;
  /** Which extensions trigger this detector. */
  extensions: string[];
  run(ctx: AssetDetectorContext): Promise<Finding[]> | Finding[];
}
```

The scan orchestrator runs `Detector[]` and `AssetDetector[]` in
two parallel passes, merges findings.

**Default asset includes/excludes:**

Include pattern set:
- Raster: `**/*.{png,jpg,jpeg,gif,webp,avif}`
- Vector: `**/*.svg`

Exclude additions on top of existing defaults:
- `**/public/vendor/**`
- `**/__snapshots__/**`
- `**/*.test.{png,jpg,jpeg,svg}` (asset fixtures used by visual tests)
- `**/fixtures/**` (already excluded by existing fixtures)

**Implication (RESOLVED).** Asset pipeline is a phase 5 task per
the plan's sequencing. No blocker. The separate-pass architecture
keeps the existing detector surface untouched.

## Investigation 4 ŌĆö Image dimension reading

**Question (┬¦9.4):** Pure-stdlib or tiny dep for reading
PNG/JPG/WebP/GIF/AVIF dimensions?

**Findings.**

Reading image dimensions only requires the file header ŌĆö none of
the pixel data. Each format has a different header layout:

- **PNG**: 8-byte signature + IHDR chunk at fixed offset; width/
  height at bytes 16-23.
- **JPG**: variable-length segments; need to scan for the SOF
  marker (multiple variants: SOF0/SOF1/SOF2/etc.) and read width/
  height from it.
- **GIF**: width/height at fixed offset (bytes 6-9).
- **WebP**: depends on subformat (VP8/VP8L/VP8X) ŌĆö three different
  header structures.
- **AVIF**: HEIF container with `ispe` box; non-trivial to parse.
- **SVG**: XML ŌĆö read width/height attributes or viewBox.

A pure-stdlib implementation is feasible but non-trivial (~400
lines covering all formats including format detection from the
magic bytes). The `image-size` library
([image-size on npm](https://www.npmjs.com/package/image-size))
covers all formats including AVIF and SVG in ~50KB, zero
dependencies, well-maintained (used by Next.js, etc.).

**Recommendation: `image-size` as a dependency on
`packages/core`.** Specifically on `core` because asset detectors
live in core, and core's dependency footprint already includes
`zod`, `fast-glob`, `ignore`, `simple-git`. Adding one more
focused, stable, single-purpose library is consistent with the
current dependency posture.

**Implication (RESOLVED).** Add `image-size` to `packages/core`
in phase 5. ~50KB to the published bundle ŌĆö acceptable.

## Investigation 5 ŌĆö Eval harness multi-agent expansion

**Question (┬¦9.3):** Add a third coding agent (Aider, Continue,
or similar) to the eval harness in 0.8.0?

**Findings.**

Existing harness invokes `claude` and `codex` CLIs in
non-interactive mode against the user's existing subscriptions ŌĆö
no API keys, no per-call billing.
[`evals/runner`](../evals/runner).

Adding a third agent has fixed cost:
- New invocation wrapper (each CLI has different flag shapes)
- New per-scenario expected-artifact rubric variance
- Setup documentation for users running evals locally

And recurring cost:
- Every new detector scenario needs to work on all three agents
- Variance sampling cost scales linearly

**Recommendation: defer to 0.9.0.** Two agents already cover the
"is this signal robust across coding agents" question. Adding a
third before we know whether the 0.8.0 detector slate is healthy
on two would expand surface area before consolidating it. Revisit
once 0.8.0 ships and we have a baseline.

**Implication (RESOLVED).** Plan ┬¦7 stays as-written ŌĆö two agents.

## Investigation 6 ŌĆö `detectors.options` schema generality

**Question (┬¦9.5):** Does the `detectors.options` shape generalise
cleanly to future detectors, or does it need more than allowlists?

**Findings.**

Reviewing the 13 kept candidates plus the existing 34 detectors,
the option-key needs that arise are:

| Option key shape | Used by |
|---|---|
| `allowedNames: string[]` | `stop_word_name`, `boolean_naming_drift` |
| `allowedLiterals: string[]` | `magic_domain_literal_scatter`, `locale_drift` |
| `allowedFunctions: string[]` | `name_behavior_mismatch` |
| `allowedExtensions: string[]` | future asset detectors |
| `thresholdBytes: number` | `oversized_raster` (specific threshold) |
| `thresholdPixels: number` | `raster_should_be_vector` |
| `excludePatterns: string[]` | many ŌĆö file-pattern exemptions |

Three categories: allowlists, thresholds, and exclusion patterns.

The `detectors.options.<id>` shape generalises cleanly:

```ts
// packages/core/src/config.ts (additions)

interface DetectorOptions {
  // Open record ŌĆö each detector validates its own keys via a
  // zod sub-schema registered at detector-load time.
  [key: string]: unknown;
}

interface CrimesConfig {
  detectors?: {
    enable?: string[];
    disable?: string[];
    /** NEW: per-detector options. */
    options?: Record<string, DetectorOptions>;
  };
}
```

Per-detector validation happens in the detector itself, not in the
top-level config schema ŌĆö this lets new detectors ship without
modifying core config validation. Unknown option keys for a known
detector raise `ConfigParseError`; options for an unknown detector
id raise the same error.

**Recommendation.** Schema is general. Ship as proposed in plan ┬¦6.

**Implication (RESOLVED).** Plan ┬¦6 design is correct.

## Investigation 7 ŌĆö `direct_date` coexistence

**Question (┬¦5.a.3):** How do the new date detectors avoid
double-counting findings that `direct_date` already covers?

**Findings.**

`direct_date` fires on every `Date.now()` or `new Date()` site in
non-test files (see
[`direct-date.ts`](../packages/core/src/detectors/direct-date.ts)).
It aggregates per-file: one finding per file containing direct
clock access.

The new date detectors operate on different patterns:

- `timezone_unsafe_parse` fires on `new Date("ŌĆ”")` with a string
  argument ŌĆö `direct_date` would also fire on this `new Date()`
  call, but for a different reason ("don't read the clock in
  domain code" vs "parsed this string in implicit local
  timezone").
- `mixed_utc_local_methods`, `locale_drift`,
  `date_equality_misuse` operate on Date *instances* ŌĆö they fire
  on `.getHours()` / `.toLocale*()` / `==` between dates, not on
  the construction site.
- `dst_naive_arithmetic`, `date_string_concat` operate on
  arithmetic / concatenation patterns ŌĆö orthogonal to construction.

**Overlap is real only for `timezone_unsafe_parse`.** Both
detectors will fire on the same line. That's actually fine ŌĆö they
give different evidence and the user fixes both with one change
(replace with timezone-aware parsing + inject a clock). No
double-counting because each detector emits one finding with
distinct charge / evidence.

**Recommendation.** Document the overlap in
`docs/finding-types/date-time.md` when it ships. Do not silence
either detector ŌĆö they're different lenses.

**Implication (RESOLVED).** No detector logic change needed.

## Investigation 8 ŌĆö Date detector prototype on real corpora

**Question (┬¦5.a.2):** What's the false-positive surface for the
six date detectors on real codebases?

**Method.** Built a throwaway AST walker at
[`.planning/prototypes/0.8.0-date-prototypes/run.ts`](./prototypes/0.8.0-date-prototypes/run.ts)
implementing all six proposed heuristics. Ran against:

- **Sanity fixture** (one intentional violation per detector) ŌĆö
  calibration baseline
- **`packages/`** (this monorepo's source ŌĆö 221 files) ŌĆö clean
  baseline
- **`evals/fixtures/02-react-dashboard`** (128 files, Next.js
  patterns)
- **`evals/fixtures/01-messy-ts-app`**, **`03-node-cli-tool`**,
  **`04-monorepo`**, **`06-stress-duplication`**,
  **`08-stress-dependency`**, **`09-clean-tiny`**,
  **`10-clean-typed`**
- **`examples/messy-ts-app`** (bundled fixture, 14 files)
- **`apps/website`**, **`evals/runner`**

**Aggregate results (587 files outside the sanity fixture):**

| Detector | Findings | False positives |
|---|---|---|
| `timezone_unsafe_parse` | 0 | 0 |
| `mixed_utc_local_methods` | 0 | 0 |
| `locale_drift` | 2 | 0 (both real `.toLocaleString()` with no locale) |
| `dst_naive_arithmetic` | 0 | 0 |
| `date_string_concat` | 0 | 0 |
| `date_equality_misuse` | 0 | 0 |

**Sanity-fixture pass rate: 5 of 6.** `date_equality_misuse`
*did not* fire on `a === b` where `a` and `b` are
`Date`-typed parameters ŌĆö the syntactic heuristic only flags
identifiers named `date`/`d`/`*At`/`*Time`/etc., which `a`/`b`
don't match. This confirms the parser-only finding from
Investigation 2: **`date_equality_misuse` genuinely needs type
info to be robust.** Without it, the detector misses the most
common form of the bug (comparing function parameters of typed
`Date`).

**Per-detector verdicts.**

- `timezone_unsafe_parse` ŌĆö **SHIP.** Pure syntactic heuristic
  works: detects `new Date(stringLiteral)` where the literal has
  no `Z` or `┬▒HH:MM` offset. Conservative enough to skip on
  numeric epoch args. FP rate observed: 0%.
- `mixed_utc_local_methods` ŌĆö **SHIP.** File-level aggregation
  catches the receiver-shared-mixed-methods pattern cleanly.
  No type info needed because the heuristic only fires when
  literal Date methods are observed on a literal receiver
  identifier. FP rate: 0%.
- `locale_drift` ŌĆö **SHIP.** Easiest detector ŌĆö `.toLocale*()`
  with zero args. 2 real findings in the React fixture confirm
  it surfaces a meaningful pattern.
- `dst_naive_arithmetic` ŌĆö **SHIP, narrow scope.** The day/hour
  constants approach (`86400000`, `604800000`, etc.) is
  conservative. Some genuine usage (e.g., `1000 * 60 * 60 * 24`)
  is captured because the prototype folds nested `*`
  expressions. Broader pattern matching (e.g., `+ 1000 * 60`
  for "add minutes") is deferred ŌĆö needs more evidence to
  calibrate.
- `date_string_concat` ŌĆö **SHIP.** Conservative
  `"ŌĆ”" + date.method()` pattern. FP rate: 0%. Real-world
  incidence: low. The detector exists more as a safety net than
  a frequent flyer.
- `date_equality_misuse` ŌĆö **SHIP v2 (with type info), DEFER v1.**
  The syntactic v1 misses too many real cases. Either:
  - (a) Defer the detector to 0.9.0 alongside `singular_plural_type_mismatch`
    v2 once we add a type checker, or
  - (b) Ship a conservative v1 that fires only on
    `new Date(...) === new Date(...)` (rare but catastrophic)
    and `dateVar === dateVar` for variables initialized to a
    Date.

  **Recommendation: (a).** v1 catches too little to justify
  the noise budget. Re-evaluate in 0.9.0.

**Implication for 0.8.0 date family.** Ship 5 of 6 detectors:
`timezone_unsafe_parse`, `mixed_utc_local_methods`,
`locale_drift`, `dst_naive_arithmetic`, `date_string_concat`.
Defer `date_equality_misuse` to 0.9.0 with the type-info work.

**Severity calibration (┬¦5.a.4 RESOLVED):**

| Detector | Default | Max | Rationale |
|---|---|---|---|
| `timezone_unsafe_parse` | medium | high | High when the string is `YYYY-MM-DD` (date-only ŌĆö almost always wrong) |
| `mixed_utc_local_methods` | high | high | Silent bug class; tests don't catch it |
| `locale_drift` | low | medium | Often intentional; medium when in user-facing output |
| `dst_naive_arithmetic` | medium | high | High when in scheduling/billing code by file path |
| `date_string_concat` | low | medium | Smell rather than always-bug |

These are starting points ŌĆö real calibration happens during the
0.8.0 dogfood phase.

## Investigation 9 ŌĆö Shape classification accuracy audit

**Question (┬¦5.c.1):** Is the existing 0.4.0ŌĆō0.6.0 shape
classifier accurate enough for `sync_io_in_hotpath` to gate on
`route_handler` / `page_export` / `react_component` / `cli_command_registrar`?

**Method.** Built a throwaway script at
[`.planning/prototypes/0.8.0-shape-audit/inspect.ts`](./prototypes/0.8.0-shape-audit/inspect.ts)
that runs the `language-js` parser against every file in a
target directory and emits the shape assigned to every function
regardless of size. Ran against the React eval fixture and the
Commander eval fixture.

**React fixture (`evals/fixtures/02-react-dashboard`):**

- 128 files, 289 functions
- 76 `react_component` ŌĆö correct (Next.js + plain React components)
- 43 `page_export` ŌĆö correct (Next.js Pages Router defaults)
- 7 `route_handler` ŌĆö correct (App Router `GET`/`POST` exports and
  Pages Router `pages/api/*.js` defaults)
- 75 `domain` ŌĆö utility/helper functions
- 88 `unknown` ŌĆö anonymous callbacks, IIFE fragments

**CLI tool fixture (`evals/fixtures/03-node-cli-tool` ŌĆö Commander.js examples):**

- 151 files, 1888 functions
- 77 `cli_command_registrar` ŌĆö correct (Commander `.action(ŌĆ”)`
  callbacks and `register*Command(program)` wrappers)
- 1091 `test_callback` ŌĆö correct (`describe`/`it`)
- 268 `domain`
- 452 `unknown`
- Zero false positives in the non-applicable shapes
  (`route_handler`, `page_export`, `react_component`) ŌĆö exactly
  what we'd want for a CLI tool

**Verdict (RESOLVED).** Shape classification is accurate enough
to gate `sync_io_in_hotpath`. Specifically:

- `route_handler` correctly catches both App Router HTTP-verb
  exports and Pages Router `pages/api/*` defaults.
- `page_export` reliably catches Next.js page/layout/template
  defaults.
- `react_component` works for PascalCase functions returning JSX.
- `cli_command_registrar` reliably catches Commander DSL functions.

The `unknown` bucket is the only category we should *not* gate
on ŌĆö anonymous callbacks are too ambiguous to assume hot-path
semantics.

**Recommended `sync_io_in_hotpath` gating shape set:** fire only
when the enclosing function shape is one of `route_handler`,
`page_export`, `react_component`, **or** `domain` (for libraries
that don't use a framework). Skip `unknown`, `test_callback`,
`cli_command_registrar` ŌĆö sync I/O in those shapes is either
intentional or test-only.

## Investigation 10 ŌĆö Policy decisions

Four research items in the plan are policy calls, not feasibility
questions. Resolved here:

### ┬¦5.b.3 ŌĆö Boolean naming React-state allowlist (RESOLVED)

The built-in allowlist for `boolean_naming_drift` ships with:

```
loading, ready, active, disabled, expanded, pending, open,
closed, visible, hidden, selected, focused, dirty, valid,
submitting, editing, dragging, hovering, checked, busy,
empty, full, online, offline, mounted, unmounted
```

These are idiomatic React-state / form-state / UI-state names
that don't carry an `is/has/should` prefix because the prefix
would be redundant ŌĆö the variable is already used as the boolean
condition (`if (loading) ŌĆ”`). Users can extend via the
`detectors.options.boolean_naming_drift.allowedNames` config.

### ┬¦5.c.2 ŌĆö `hardcoded_local_path` scope (RESOLVED)

Flag these patterns in non-test, non-script code:

- `/Users/<name>/ŌĆ”` (macOS)
- `/home/<name>/ŌĆ”` (Linux ŌĆö careful not to match `/home` alone)
- `C:\\Users\\<name>\\ŌĆ”` (Windows, double-escaped)
- `~/<ŌĆ”>` is also user-specific but is often intentional in
  scripts ŌĆö defer to v2

Skip:

- `/tmp/<ŌĆ”>` ŌĆö legitimate transient file ops are common
- `/var/<ŌĆ”>`, `/etc/<ŌĆ”>` ŌĆö system paths, intentional when used
- `/usr/local/bin/<ŌĆ”>` ŌĆö also intentional

Skip entire files matching `**/{scripts,examples,test,tests,__tests__,fixtures}/**`.

### ┬¦5.c.3 ŌĆö `hardcoded_localhost` scope (RESOLVED)

Flag `localhost:NNNN`, `127.0.0.1:NNNN`, `0.0.0.0:NNNN` in
non-test, non-config code.

Skip files matching:

```
**/.env*, **/docker-compose*.{yml,yaml},
**/Dockerfile*, **/*.config.{js,ts,mjs,cjs,json,yaml,yml},
**/{scripts,examples,test,tests,__tests__,fixtures,docs}/**,
**/README*.md, **/CHANGELOG*.md
```

The `scripts/` exclusion is important ŌĆö dev scripts routinely
hardcode localhost. The `*.config.*` exclusion covers Vite,
Next.js, Webpack, etc. configs that legitimately reference dev
servers.

### ┬¦5.d.2 ŌĆö `oversized_raster` thresholds (RESOLVED)

## Pushback ŌĆö additional considerations on resolved items

Reviewing the earlier "RESOLVED" items critically before any of
them harden into code:

### On Investigation 2 (no ts-morph dep, syntactic v1)

The prototype confirmed (Investigation 8) that
`date_equality_misuse` cannot ship in v1 without type info. By
extension, `singular_plural_type_mismatch` will have the same
shape ŌĆö its syntactic v1 will miss cases where the type
annotation is a typedef'd alias (`type UserId = string`) or
imported (`import { User } from "./types"`).

**Caveat to record in the plan:** the syntactic v1 of
`singular_plural_type_mismatch` will only fire on declarations
where the type annotation is a *literal* identifier matching the
expected singular/plural form. Aliased and re-exported types are
silently skipped. This is acceptable for 0.8.0 *if* we are
explicit about the false-negative rate in the docs.

If the FP rate turns out to be high, we should defer the detector
to 0.9.0 with proper type info via the raw `ts` API (still no
ts-morph dep). Add this as an explicit gate in phase 3
sequencing.

### On Investigation 3 (separate asset pass)

Asset findings will have meaningless `blast_radius` (image isn't
imported), `test_gap` (image isn't tested), and `churn` scores
unless the scoring system gracefully handles "this finding has no
applicable scoring data."

The scoring layer in `packages/core/src/scoring/build.ts` will
need to either:

- Default missing signals to 0 (current behaviour for files with
  no git history), or
- Add an explicit `Finding.scoring_applicable?: boolean` so
  consumers know not to rank by these signals.

**Recommendation:** default to 0 ŌĆö keep the surface uniform.
Document that asset findings always have these scores at 0;
asset detectors should set their own `severity` and `confidence`
to carry the signal. No schema change needed.

### On Investigation 4 (`image-size` dep)

`image-size` parses untrusted binary data ŌĆö that's attack surface.
For a CLI tool reading files the user already trusts (their own
repo), this is low risk. But:

- Add `image-size` to a tight version pin and Renovate-track it
- Restrict the asset detector pass to extensions in the allowlist
  (don't open arbitrary binary files)
- Catch parse errors gracefully ŌĆö a malformed image should produce
  no finding, not crash the scan

These are implementation-time concerns, not blockers.

### On Investigation 6 (`detectors.options` open record)

The "open record where each detector validates its own keys"
pattern means a typo in a detector option key (`allowdNames`
instead of `allowedNames`) only surfaces when that detector
runs ŌĆö not at config-load time. Users may not notice the typo
until they're surprised by an unexpected finding.

**Mitigation:** require detectors to register their option schema
with the core config loader at module-load time. The top-level
`CrimesConfigSchema` then knows every valid detector id and its
expected options shape. This catches typos at config load.

Slightly more plumbing than the original proposal, but worth it.
Updated design:

```ts
// packages/core/src/detector.ts (additions)

export interface DetectorOptionSchema<T = unknown> {
  detectorId: string;
  schema: z.ZodType<T>;
}

export interface Detector {
  // ...existing fields...
  /** NEW: optional zod schema for `detectors.options.<id>`. */
  optionsSchema?: z.ZodType<unknown>;
}
```

Core's config loader collects all `detector.optionsSchema` from
the built-in detector list and synthesizes the per-detector
sub-schemas. Unknown detector ids in `detectors.options` raise
`ConfigParseError`.

## Investigation 11 ŌĆö `stop_word_name` exports-only prototype

**Question (┬¦5.b.4):** Does scoping `stop_word_name` to *exported*
identifiers produce a useful agent-risk signal that ESLint's
`id-denylist` does not?

**Method.** Built a prototype at
[`.planning/prototypes/0.8.0-stop-word-exports/run.ts`](./prototypes/0.8.0-stop-word-exports/run.ts)
that walks every TS/JS file, collects exported declarations
(function, class, variable, type alias, interface) at top level,
and flags any whose name matches a stop-word allowlist:
`data, info, obj, item, thing, stuff, value, result, temp, tmp,
x, y, z, handle, process, manage, perform, Manager, Handler,
Helper, Util, Utils, Wrapper, Worker, Service, Factory, Maker,
Processor`.

**Results:**

| Corpus | Files | Findings |
|---|---|---|
| `packages/` | 221 | 0 |
| `evals/fixtures/02-react-dashboard` | 128 | 0 |
| `evals/fixtures/03-node-cli-tool` | 151 | 0 |

**Interpretation.** The signal doesn't materialize at the export
level. Real codebases rarely export top-level identifiers named
`data` / `Manager` / `Helper`. Where stop-word names *do* appear,
it's usually as:

- Function parameters (`function process(data: User[])`)
- Local variables (`const temp = x.transform()`)
- Properties in object types

These are NOT exports. The exports-only filter is too narrow ŌĆö
it filters out exactly the case the detector was meant to catch.

**Verdict (RESOLVED).** **Defer `exported_stop_word_name` to
0.9.0+.** Two paths to revisit:

1. Broaden the scope to parameters and locals (loses the
   exports-angle distinction from `id-denylist`)
2. Drop the detector entirely (ESLint covers the broader scope
   sufficiently; we add no new lens)

Either way, this isn't a 0.8.0 detector. Naming-tier ships with
just `singular_plural_type_mismatch` and `boolean_naming_drift`.

## Investigation 12 ŌĆö `catch_rethrow_no_context` frequency

**Question (┬¦5.c.5):** Is the catch-rethrow-without-context
pattern common enough in modern TS code to justify a dedicated
detector?

**Method.** Grep across the same 500+ file corpus for any
`catch (ŌĆ”) { ŌĆ” throw ŌĆ” }` pattern.

**Results:** **0 matches** across `packages/`, `evals/fixtures/`,
and `examples/`.

**Interpretation.** Modern TS codebases either:

- Let exceptions propagate (no `catch` at all ŌĆö most common)
- Catch, augment with context, then rethrow (`throw new Error("ŌĆ”", { cause: e })` or framework-specific wrapping)
- Catch and convert to a Result type
- Catch and log without rethrowing

The "catch and bare-rethrow" anti-pattern is rare enough to be
non-actionable as a default detector. ESLint's `no-useless-catch`
already catches the most egregious case.

**Verdict (RESOLVED).** **Drop `catch_rethrow_no_context` from
the 0.8.0 candidate list.** Not a useful lens.

## Investigation 13 ŌĆö Pluralisation library scope

**Question (┬¦5.b.2):** Hand-rolled pluralisation or add a
`pluralize` library dependency?

**Decision (RESOLVED): hand-rolled, ~50 lines.** The cases
`singular_plural_type_mismatch` actually needs to handle are
narrow:

```
cat   ŌåÆ cats     (default: + s)
box   ŌåÆ boxes    (ends in s/x/z/ch/sh: + es)
party ŌåÆ parties  (consonant + y: y ŌåÆ ies)
day   ŌåÆ days     (vowel + y: + s)
status ŌåÆ statuses
class  ŌåÆ classes
```

Plus a small irregular list:

```
child ŌåÆ children, person ŌåÆ people, man ŌåÆ men, woman ŌåÆ women,
foot ŌåÆ feet, tooth ŌåÆ teeth, mouse ŌåÆ mice, criterion ŌåÆ criteria,
datum ŌåÆ data, medium ŌåÆ media, analysis ŌåÆ analyses
```

Plus uncountables that should NEVER plural-mismatch (always
treat as both singular and plural):

```
data, information, news, software, staff, equipment,
audio, video, traffic, weather
```

Total surface: a small lookup table + 3 regex rules. ~60 lines
including JSDoc. No dependency footprint, no licensing question,
fully understandable in code review.

The `pluralize` library (~10KB on npm) handles more obscure cases
(`octopus` ŌåÆ `octopi`) but those don't appear in identifier
naming. The hand-rolled version handles 99%+ of real code.

## Investigation 14 ŌĆö Dead export agent-risk variant

**Question (┬¦5.c.4):** Is there a useful agent-risk signal from
"dead export whose name is edit-distance 1 from a live export"?

**Approach considered.** Build a prototype that:

1. Identifies all exports across the workspace
2. Identifies which are imported anywhere
3. For each dead export, compute Levenshtein distance to every
   live export in the same module
4. Flag if distance is 1-2 and the live export shares a substantial
   prefix

**Decision (RESOLVED): defer to 0.9.0.** Reasoning:

- The hypothesis ("agents pick wrong-but-close names") is
  *plausible* but unproven. Need real evidence (e.g., observation
  in eval harness runs) before committing to detector design.
- Knip / ts-prune already give users the broader "dead exports
  exist" signal ŌĆö our novelty would need to be specifically the
  edit-distance agent-risk angle, which deserves its own
  research cycle.
- 0.8.0 is already shipping 13 detectors + a new config feature.
  Adding speculative work increases scope without strengthening
  the slate.

Track as a 0.9.0 candidate. Re-evaluate once we have agent
eval-run evidence that the pattern actually causes confusion.
